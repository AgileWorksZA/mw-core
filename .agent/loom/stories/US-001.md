# US-001: Implement Product Canonical Entity with Repository Pattern

## Why (Root Motivation)

Products are one of the core MoneyWorks entities essential for completing the canonical domain model. They're required for transaction creation, product catalog management, and pricing operations. Completing this entity set provides full coverage of primary MoneyWorks tables, enabling comprehensive accounting integrations and unlocking downstream features like purchase order generation, inventory tracking, and sales forecasting.

## Description

Complete the MoneyWorks canonical entity set by implementing the Product entity with full parity to Name and Tax Rate entities. This includes:

- **Canonical type definitions** preserving exact MoneyWorks field names (StockCode, Description, SellPrice, CostPrice, etc.)
- **Branded types** for type safety (ProductCode)
- **Complete field metadata** with validation rules
- **ProductRepository** extending BaseMoneyWorksRepository with read-only queries
- **Specialized query methods** for common product operations
- **Multi-format export support** (compact, compact-headers, full, schema)
- **Field discovery integration** following the XML → TSV pattern

## Acceptance Criteria

- [ ] **AC-001**: Canonical types defined with exact MoneyWorks Product field names (StockCode, Description, SellPrice, CostPrice, Unit, Category, Location, etc.) in `@moneyworks/canonical/entities/products/types.ts` matching ProductCode branded type from utilities

- [ ] **AC-002**: Product enums implemented for ProductType, Status, and other enumerated fields with canonical MW values in `enums.ts` with bit-mapped flags for Products (e.g., NonStock, Discontinued, Kit)

- [ ] **AC-003**: Complete field metadata in `fields.ts` defining all Product fields with MoneyWorksProductField interface including: fieldName, dataType (T/N/D/B/S), maxLength, description, required flag, indexed status, relatedEntity, and validationRules

- [ ] **AC-004**: Validator functions in `validators.ts` implementing: validateProduct() for business rules, validateProductCode() for stock code format, validatePricing() for price constraints, checkProductStatus() for active/discontinued logic, and helper functions like isKit(), isNonStock(), getProductMargin()

- [ ] **AC-005**: ProductRepository in `@moneyworks/data/repositories` extending BaseMoneyWorksRepository<MoneyWorksProduct> with prepare() and postProcess() methods handling numeric field conversion, boolean parsing, and branded type conversion

- [ ] **AC-006**: Read-only repository operations implemented: findByKey(stockCode), findAll(params), find(search?, params?) supporting MoneyWorks search expressions with proper error handling and mock mode fallback

- [ ] **AC-007**: Specialized query methods implemented: getActiveProducts(), getByCategory(category), getByLocation(location), getKitProducts(), getNonStockProducts(), search(text, options), getByPricingLevel(level), and getWithCurrentCosts()

- [ ] **AC-008**: Multi-format export support for all queries returning: compact (arrays), compact-headers (with headers), full (objects), schema (with metadata) via client.smartExport() integration

- [ ] **AC-009**: Field discovery integration using XML export to dynamically determine Product table structure and field ordering, with TSV parsing following field discovery pattern, caching discovered structure for performance

- [ ] **AC-010**: Barrel exports in `index.ts` re-exporting types, enums, field definitions, validators, and repository with MW_PRODUCT_TERMS constant for canonical terminology (e.g., STOCK_CODE, SELL_PRICE, COST_PRICE)

- [ ] **AC-011**: Unit tests in `product.repository.test.ts` covering: findByKey returns correct product, findAll respects pagination, specialized queries filter correctly, data type conversions work, error handling for invalid codes, and mock mode fallback

- [ ] **AC-012**: Type safety verified with TypeScript compiler - all fields strongly typed with no any casts, branded types prevent misuse, MoneyWorksProduct interface correctly extends from field metadata

## Weave Knowledge

### Patterns Applied
- **E:canonical-dsl-pattern** - Preserve exact MoneyWorks terminology (StockCode, not productCode)
- **E:field-discovery-pattern** - Use XML exports to discover field structure before TSV parsing
- **E:branded-types-pattern** - ProductCode branded type for compile-time safety
- **E:multi-format-export-pattern** - Support 4 export formats (compact, compact-headers, full, schema)

### Best Practices
- **Q:preserve-mw-terminology-workflow** - NEVER translate MoneyWorks field names
- **Q:field-discovery-workflow** - XML first for structure discovery, then parse TSV efficiently

### Pain Points to Avoid
- TSV format lacks headers - must use XML export for field discovery first
